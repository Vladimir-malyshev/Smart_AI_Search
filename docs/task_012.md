Глубокий архитектурный рефакторинг: мы переводим сервис в режим Tool Retrieval API, выносим LLM-промпты во внешний конфиг и добавляем сохранение артефактов на стороне тестового скрипта.

Пожалуйста, выполни следующие шаги:

1. Внешний конфиг промптов (app/config/prompts.yaml)

    Создай директорию app/config/ и файл prompts.yaml.

    Перенеси туда SYSTEM_PROMPT из snippet_evaluator.py (назови ключ snippet_evaluator_system).

    Напиши новый промпт для Судьи под ключом ai_judge_system. Роль Судьи теперь — Куратор (Data Curator). Его задача — оценить усеченные тексты. Если информации достаточно (status: "complete"), он должен вернуть список релевантных ссылок. Обязательно добавь правило: "Исключай дубликаты. Если источник А содержит ту же информацию, что и источник Б, но полнее — возвращай только А. Если дополняют друг друга — возвращай оба".

2. Обновление модулей для чтения конфига (app/modules/snippet_evaluator.py и app/modules/ai_judge.py)

    Добавь чтение соответствующих промптов из app/config/prompts.yaml (используй библиотеку pyyaml). Удали жестко зашитые тексты промптов из кода.

3. Перепрошивка Судьи (app/modules/ai_judge.py)

    В Pydantic-модели JudgeOutput удали поле final_answer и добавь поле useful_urls: List[str].

    Убедись, что если status: "incomplete", useful_urls возвращается как пустой список [].

4. Изменение формата выдачи API (app/main.py)

    В run_research_pipeline, когда judge_result.status == "complete", скрипт больше не должен возвращать сгенерированный текст.

    Пройдись по списку judge_result.useful_urls. Достань полные нетронутые тексты из accumulated_context и склей их в одну Markdown-строку (разделяй маркерами ### Источник: [URL]).

    Верни эту склеенную строку в поле "answer" словаря результата. В консольный лог (logger.info) выводи только короткое саммари (статус, итерации, количество источников). Никакого сохранения файлов на стороне API!

5. Умное логирование в тестах (скрипт тестирования, например test_run.py)

    В скрипте, который запускает тесты, добавь создание директории logs/results/ (если её нет).

    После получения успешного ответа от API, сохраняй содержимое поля answer (ту самую Markdown-портянку) в отдельный файл. Назови файл так, чтобы было понятно, к какому запросу он относится (например, logs/results/test_01_pogoda_v_moskve.md или используя timestamp).

Изменяй только необходимые файлы. Убедись в правильной обработке путей для YAML-файла.
