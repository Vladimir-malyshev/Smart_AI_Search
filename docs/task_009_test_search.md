Шаг 1. Конвертация тестовых данных

    Прочитай файл tests/Search test/test_001.md, в котором лежат текстовые описания тестовых запросов (простые, средние, сложные).

    Спарси эти данные и создай структурированный файл tests/test_cases.json. В нем должен быть массив объектов, где каждый объект содержит поля: level (уровень сложности), query (запрос) и goal (цель).

Шаг 2. Разработка скрипта автоматического тестирования

    Создай исполняемый файл tests/run_automated_tests.py.

    Скрипт должен асинхронно читать test_cases.json и по очереди отправлять POST-запросы к нашему локальному API-эндпоинту (/api/v1/research).

    Критическое правило (Анти-бан): После завершения каждого запроса (неважно, успешного или с ошибкой) скрипт обязан делать паузу ровно в 5 секунд (asyncio.sleep) перед отправкой следующего.

Шаг 3. Глубокое логирование и отчетность

    Скрипт-тестировщик должен не просто выводить финальный ответ, но и вести подробный лог (в консоль и/или в отдельный файл отчета).

    Убедись, что глобальная конфигурация logging в проекте настроена так, чтобы в консоли при прогоне тестов мы четко видели внутреннюю «кухню» сервиса:

        Сколько сырых сниппетов вернул поисковик (SearXNG).

        Какие конкретно URL выбрал Snippet Evaluator (LLM).

        Какой статус (complete или incomplete) и новые запросы сгенерировал AI-Судья на каждой итерации.

    По итогу прогона всех тестов скрипт должен вывести краткую сводку: сколько тестов прошло успешно, среднее время выполнения и среднее количество потраченных итераций.
