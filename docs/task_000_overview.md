# Автономный AI-исследователь (Deep Research Service)

## Что это за продукт

Это локальный веб-сервис, который выполняет роль автономного исследователя: принимает вопрос или задачу от пользователя (или другого AI-агента), самостоятельно формирует поисковые запросы, собирает данные из интернета, читает релевантные страницы и синтезирует структурированный ответ.

По функциональности аналог Perplexity Deep Research или Google Deep Research, но без зависимости от платных поисковых API.

---

## Ключевая идея

Вместо платных API поиска (Google, Bing) система использует публичную сеть серверов **SearXNG** — это децентрализованный открытый метапоисковик. Серверов в сети сотни, они бесплатны, но нестабильны: часть из них падает, тормозит или блокирует запросы. Чтобы система была надёжной, она сама управляет репутацией этих серверов и маршрутизирует запросы только на живые и быстрые ноды.

Поверх поискового слоя работает **agentic RAG-петля** на базе LLM (Google Gemini): модель планирует запросы, отбирает полезные страницы, читает их полный текст и решает — достаточно ли собранных данных для ответа, или нужен ещё один круг поиска.

---

## Инфраструктура и деплой

Сервис разрабатывается в два этапа:

**Этап 1 — Разработка:**
- Приложение запускается локально
- Redis (хранилище репутации нод) установлен на удалённом сервере
- Подключение к Redis осуществляется через SSH-туннель
- Строка подключения: `redis://localhost:{LOCAL_TUNNEL_PORT}`
- Пример команды туннеля: `ssh -L 6379:localhost:6379 user@remote-server`

**Этап 2 — Продакшн:**
- Всё живёт на одном сервере: FastAPI-приложение + Redis + фоновые воркеры
- Строка подключения к Redis меняется на: `redis://localhost:6379`
- Никаких туннелей, никаких внешних зависимостей по сети

Конфигурация подключения к Redis **вынесена в переменную окружения** `REDIS_URL`, чтобы переключение между режимами не требовало изменений в коде.

## Архитектура: 8 модулей

| Задача | Модуль | Роль |
|--------|--------|------|
| 1 | Redis State Manager | Хранит репутацию SearXNG-нод, управляет карантином |
| 2 | SearXNG Harvester | Фоновый воркер: находит живые ноды и добавляет в Redis |
| 3 | Smart Execution Engine | Параллельная отправка поисковых запросов по нодам |
| 4 | Query Expansion (LLM) | Превращает запрос пользователя в 3-4 машинных поисковых запроса |
| 5 | Snippet Evaluator (LLM) | Отбирает 2-4 URL из сырой поисковой выдачи |
| 6 | Jina Reader | Скачивает полный текст страниц в Markdown через r.jina.ai |
| 7 | AI-Судья (LLM) | Оценивает собранный контекст и решает: отвечать или искать дальше |
| 8 | Master Orchestrator + FastAPI | Собирает все модули в пайплайн, выставляет REST API |

---

## Поток данных (Happy Path)

```
POST /api/v1/research {"query": "...", "goal": "..."}
    │
    ├── [Задача 4] LLM генерирует 3-4 поисковых запроса
    │
    ├── [Задача 3] Параллельный поиск по топовым SearXNG-нодам из Redis
    │       └── Репутация нод обновляется по результатам (быстро = +, медленно = -, ошибка = карантин)
    │
    ├── [Задача 5] LLM выбирает 2-4 наиболее релевантных URL
    │
    ├── [Задача 6] Jina читает полный текст страниц → Markdown (с обрезкой по лимиту)
    │
    ├── [Задача 7] LLM-Судья оценивает контекст
    │       ├── status: "complete" → возвращаем ответ
    │       └── status: "incomplete" → генерируем новые запросы → возвращаемся к Задаче 3
    │
    └── Ответ: синтезированный текст + источники
```

Максимум итераций: **3**. Глобальный таймаут: **45 секунд**.

## Используемые технологии

- **Python 3.12**, AsyncIO
- **FastAPI** — REST API и управление жизненным циклом сервиса
- **Redis** (redis.asyncio) — репутационное хранилище нод
- **aiohttp** — асинхронные HTTP-запросы к SearXNG и Jina
- **Google Gemini API** с моделью **Gemma 3** (старшая из линейки) — LLM для задач 4, 5, 7
- **Jina Reader API** (r.jina.ai) — извлечение чистого текста страниц

---

## Важные принципы разработки

1. **Конфигурация через env-переменные**: `REDIS_URL`, `GEMINI_API_KEY`, `JINA_API_KEY`, `MAX_ITERATIONS`, `GLOBAL_TIMEOUT_SEC`
2. **Нет глобальных переменных состояния**: каждый запрос к API работает со своим изолированным контекстом
3. **Graceful degradation**: ни один модуль не должен падать и останавливать всю систему — только логировать ошибку и продолжать работу с тем, что есть
4. **Атомарные операции в Redis**: все изменения счётов через `ZINCRBY`, без read-modify-write
5. **Туннель прозрачен для кода**: приложение всегда видит Redis как `localhost`, независимо от окружения
