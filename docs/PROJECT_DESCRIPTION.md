# Smart AI Search — описание проекта

## 1. Цель и концепция

**Smart AI Search** — это локальный веб‑сервис для автономного поиска и синтеза информации, реализованный как REST API. Сервис принимает от клиента пользовательский запрос и описание цели исследования, а в ответ возвращает структурированный набор текстовых материалов из открытых веб‑источников, релевантных этому запросу.

Ключевая идея системы — не просто найти и вернуть ссылки, а **провести полноценный исследовательский цикл**: самостоятельно оценить, достаточно ли собранной информации для достижения цели, и если нет — автоматически переформулировать запрос и продолжить поиск. Это делает систему ближе к понятию «агентного» ИИ, нежели к классическому поисковику или обёртке над API.

Сервис спроектирован как **Tool Retrieval API**. Он не генерирует итоговый ответ пользователю — он возвращает сырые, извлечённые тексты из источников. Финальный синтез ответа намеренно вынесен за пределы сервиса и может выполняться любой LLM на стороне потребителя. Это позволяет использовать сервис как универсальный «инструмент» в более широких AI-агентных пайплайнах.

## 2. Архитектура и технологический стек

Сервис построен на **Python 3.12** с использованием **AsyncIO** повсеместно: каждая операция ввода-вывода (запросы к API, чтение страниц) выполняется асинхронно, что позволяет проводить параллельный поиск по нескольким запросам одновременно без блокировки.

**Технологический стек:**

- **FastAPI + Uvicorn** — HTTP-сервер. Один эндпоинт: `POST /api/v1/research`. Параметры: `query` (текст запроса) и `goal` (цель исследования). Ответ включает статус, набор извлечённых текстов, список использованных URL, количество итераций и время выполнения.
- **aiohttp** — асинхронный HTTP-клиент для всех внешних запросов (к поисковым API и ридерам страниц).
- **google-genai** — SDK для работы с Google Gemini/Gemma. Единственный используемый LLM-провайдер в текущей конфигурации.
- **python-dotenv** — вся конфигурация вынесена в переменные окружения (`.env`). Никаких хардкодных значений в коде.
- **pyyaml** — системные промпты для LLM вынесены в конфигурационный файл `app/config/prompts.yaml`, что позволяет менять инструкции для моделей без изменения кода.

## 3. Пайплайн исследования (Research Pipeline)

Основная логика сосредоточена в `app/main.py` в функции `run_research_pipeline()`. Это итеративный RAG-цикл (Retrieval-Augmented Generation), состоящий из пяти этапов, которые повторяются до трёх раз (параметр `MAX_ITERATIONS`).

### Этап 1: Формирование поисковых запросов

На **первой итерации** используется точный запрос пользователя без изменений — это принципиальное архитектурное решение, позволяющее избежать потери смысла при раннем «расширении» запроса.

На **последующих итерациях** новые поисковые запросы генерируются AI-Судьёй (см. ниже) на основе анализа пробелов в уже собранных данных. Модуль `query_expansion.py` также реализует самостоятельное расширение запросов через LLM, хотя в текущей конфигурации эта функция остаётся в резерве.

### Этап 2: Параллельный поиск (Execution Engine)

Модуль `execution_engine.py` принимает список запросов и выполняет их **параллельно** через `asyncio.gather()`. Результат — плоский список поисковых сниппетов (заголовок, URL, краткое описание страницы).

Ключевая особенность — **провайдер-агностичная архитектура**: выбор поискового бэкенда определяется переменной окружения `SEARCH_PROVIDER`. На текущий момент реализованы:
- **Tavily** (основной провайдер) — REST API, специализированный для задач AI-агентов. Возвращает структурированные результаты с хорошим качеством сниппетов.
- **Jina Search** (резервный) — поиск через прокси `s.jina.ai`.
- **SearXNG** (заглушка, в разработке) — децентрализованный поисковый движок с самохостингом.

После сбора результатов выполняется **дедупликация по URL** и фильтрация по доменам (соцсети, агрегаторы) и типам файлов (PDF, DOCX и др.).

### Этап 3: Оценка сниппетов (Snippet Evaluator)

Прежде чем тратить ресурсы на полное чтение страниц, LLM-модель (`EVALUATOR_LLM_MODEL`, в конфигурации — `gemma-4-26b-a4b-it`) анализирует заголовки и сниппеты найденных результатов и выбирает **не более 4 наиболее релевантных URL** для цели исследования.

Реализована **защита от галлюцинаций**: все возвращённые моделью URL сверяются с фактическим списком входных сниппетов. Ссылки, которых не было во входных данных, автоматически отбрасываются с логированием.

### Этап 4: Извлечение контента (Jina Reader)

Модуль `jina_reader.py` запрашивает полный текст страниц через сервис `r.jina.ai`, который рендерит JavaScript-сайты и конвертирует HTML в чистый Markdown. Запросы выполняются параллельно с ограничением одновременных соединений (`JINA_CONCURRENCY`, по умолчанию 4).

Реализованы:
- **Фильтрация шума**: через заголовки `X-Remove-Selector` передаётся список CSS-элементов для удаления (навигация, реклама, комментарии).
- **Защита контекста**: текст каждой страницы усекается до `JINA_MAX_CHARS` (20 000 символов) с маркером об усечении.
- **Эвристика блокировок**: автоматически распознаёт и отбрасывает ответы с заглушками Cloudflare, paywall-страницами, страницами с капчей.
- Работает **без API-ключа** в публичном режиме (с ограничениями). При HTTP 451 (ограничения прав) страница пропускается, пайплайн продолжает работу.

Извлечённые тексты накапливаются в словаре `accumulated_context` (URL → текст), который **кумулятивно пополняется** с каждой итерацией.

### Этап 5: AI-Судья (AI Judge)

Ключевой интеллектуальный компонент сервиса — модуль `ai_judge.py`. Большая языковая модель (`JUDGE_LLM_MODEL`, в конфигурации — `gemma-4-31b-it`) получает весь накопленный контекст и принимает одно из двух решений:

- **`complete`** — информации достаточно. Модель возвращает список `useful_urls` — URL, из которых складывается финальный ответ.
- **`incomplete`** — информации не хватает. Модель возвращает `missing_info` (описание пробелов) и `new_queries` (новые поисковые запросы для следующей итерации), которые не дублируют уже выполненные.

Для защиты от бесконечного цикла и экономии ресурсов: контекст перед передачей Судье **усекается до 6 000 символов на источник** (`JUDGE_MAX_CHARS_PER_SOURCE`). Полные тексты хранятся в `accumulated_context` и попадают в финальный ответ без урезки.

На последней итерации Судья принудительно переводится в режим «сохранить лучшее из собранного» — это предотвращает ситуацию, когда система возвращает пустой ответ на трудный запрос.

## 4. Абстрактный слой LLM

Модуль `app/core/llm.py` реализует единый интерфейс для генерации JSON-ответов от языковых моделей. Бизнес-логика нигде не импортирует SDK провайдеров напрямую — только через этот слой.

Поддерживаются **два провайдера** (`LLM_PROVIDER`): Gemini (Google genai SDK) и OpenAI-совместимые API (включая локальные через `OPENAI_BASE_URL`).

Реализована **таблица возможностей модели** (`MODEL_CAPABILITIES`): системы вроде `gemma-4` поддерживают нативный JSON-режим (`response_mime_type=application/json`) и прямой JSON Schema, тогда как старые Gemma 3 работают через принудительные инструкции в промпте с последующим парсингом. Переключение происходит автоматически по префиксу имени модели.

При ошибке **429 RESOURCE_EXHAUSTED** реализован экспоненциальный backoff с поддержкой извлечения задержки из ответа API (`retryDelay`), с максимальным числом попыток `LLM_MAX_RETRIES`.

## 5. Надёжность и отказоустойчивость

**Graceful Degradation** — принцип, пронизывающий всю систему: ни один модуль не останавливает весь пайплайн при ошибке. Если поиск вернул 0 сниппетов — цикл продолжается. Если Jina не смог прочитать URL — он пропускается, остальные обрабатываются. Если LLM вернул невалидный JSON — применяется regex-парсинг как резервный метод.

**Изоляция состояний**: каждый запрос к `/api/v1/research` создаёт собственный изолированный контекст (`accumulated_context`, `all_executed_queries`). Никаких разделяемых глобальных переменных между запросами.

**Глобальный таймаут** (`GLOBAL_TIMEOUT_SEC`, 600 секунд) оборачивает весь пайплайн через `asyncio.wait_for()` и гарантирует, что клиент всегда получит ответ, даже если какой-то LLM-вызов завис.

**E2E тестирование**: в `tests/run_automated_tests.py` реализован автономный тест-раннер, который прогоняет наборы тестовых кейсов разной сложности (easy/medium/hard), замеряет время, фиксирует количество итераций и сохраняет результаты в `logs/results/`. После каждого успешного теста отдельная LLM-модель синтезирует краткий ответ на основе собранных данных — это позволяет оценить реальное качество контента.

## 6. Текущая конфигурация и состояние

Система работает на связке **Tavily Search + Gemma 4** (семейство Google). Поиск выполняется через REST API Tavily, чтение страниц — через Jina Reader в публичном режиме. Все LLM-вызовы работают через Google Gemini API:

| Роль | Модель |
|---|---|
| Оценщик сниппетов | `gemma-4-26b-a4b-it` |
| AI-Судья | `gemma-4-31b-it` |

Вся конфигурация управляется через `.env`: ни одного хардкодного значения в бизнес-логике. Модель, провайдер, ключи, таймауты, лимиты контекста, пороги отбора — всё это переменные окружения, задокументированные в `.env.example`.
