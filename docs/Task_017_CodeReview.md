## Раздел 1: Критические проблемы (блокеры)

---

### 1.1 `redis_manager.py`: Создание подключения к Redis на уровне модуля

**Файл:** `app/modules/redis_manager.py`, строки 12–20

**Что плохо:**  
`ConnectionPool` и `redis_client` создаются прямо при импорте модуля. Переменная `REDIS_URL = os.environ["REDIS_URL"]` вызывает `KeyError` при старте, если переменная не задана, — сервер просто не поднимается. Более серьёзно: сетевое подключение устанавливается ещё до того, как FastAPI инициализировал `lifespan`-контекст, что нарушает принцип явного управления жизненным циклом ресурсов.

**Почему это проблема:**  
В тестах приходится через `os.environ["REDIS_URL"] = ...` патчить переменную до любого импорта из `app.*`, иначе тест рухнет при импорте. Это хрупкая конструкция. Кроме того, если Redis недоступен в момент старта, создание пула упадёт с исключением, а не переключится в fallback-режим.

**Как исправить:**  
Перенести создание пула и клиента из уровня модуля в функцию-фабрику с ленивой инициализацией (паттерн lazy singleton). Функция должна проверять наличие `REDIS_URL` и перехватывать исключение соединения, устанавливая `_redis_available = False`. Идеально — инициализировать подключение в `lifespan` приложения и передавать клиент через состояние приложения (`app.state`).

### 1.2 `search_router.py`: Дублирующий «зомби-модуль»

**Файл:** `app/modules/search_router.py`

**Что плохо:**  
Файл `search_router.py` содержит полную альтернативную реализацию поискового роутера с собственными провайдерами (`jina`, `ddg`, `searxng`) и собственным классом `SearchSnippet`. Этот класс **полностью дублирует** `SearchSnippet` из `execution_engine.py`. В `app/main.py` импортируется только `execution_engine`, `search_router` нигде не используется. Он фактически «мёртвый» код, который продолжает существовать в кодовой базе.

**Почему это проблема:**  
Любой разработчик, открывший проект, будет дезориентирован: есть два модуля с одинаковыми обязанностями. Два класса `SearchSnippet` с разными сигнатурами (в `search_router.py` в фикстурах тестов даже видно поле `source_node`, которого нет в актуальном датаклассе) создают несоответствие реального кода и тестов. `search_router.py` также тянет лишние зависимости: `beautifulsoup4` и `duckduckgo-search` используются только там, хотя фактически весь поиск идёт через `execution_engine.py`.

**Как исправить:**  
Удалить `search_router.py` полностью. Если DuckDuckGo-поиск нужен как запасной вариант — реализовать его внутри `execution_engine.py` как ещё один провайдер, аналогично уже существующим `_search_jina` и `_search_tavily`. Почистить `requirements.txt` от `beautifulsoup4` и `duckduckgo-search`, если они больше нигде не используются.

### 1.3 `tests/test_main.py`: Тесты написаны под старый интерфейс

**Файл:** `tests/test_main.py`

**Что плохо:**  
Тесты компрометируют подход к тестированию на нескольких уровнях:

1. Мок `mock_execute.return_value` создаёт `SearchSnippet(url=..., title=..., snippet=..., source_node="n")` — с полем `source_node`, которого нет в актуальном датаклассе `SearchSnippet` из `execution_engine.py`. Тест буквально описывает старую версию кода.
2. `JudgeOutput(status="complete", final_answer="Mocked Answer", ...)` — поле `final_answer` не существует в актуальном `JudgeOutput`. Актуальная модель возвращает `useful_urls` (список URL), а не готовый текст ответа.
3. В моке патчится `app.main.expand_query` и `app.modules.harvester.harvester_loop`, которых в текущем `app/main.py` нет — они были удалены в ходе рефакторинга.

**Почему это проблема:**  
При запуске `pytest` эти тесты упадут с `TypeError` при создании мок-объектов. Получается, что тестовое покрытие на самом деле нулевое: тесты не проверяют код, они сломаны. Это создаёт ложное ощущение безопасности.

**Как исправить:**  
Полностью переписать `test_main.py` под актуальный интерфейс: убрать `source_node` из `SearchSnippet`, убрать `final_answer` из `JudgeOutput` (заменить на `useful_urls`), убрать патчи несуществующих функций. Аналогичная проверка нужна для `test_ai_judge.py` — там тоже используется `final_answer` и `result.final_answer`, которых нет в `JudgeOutput`.

## Раздел 2: Серьёзные проблемы (высокий приоритет)

---

### 2.1 `jina_reader.py`: Конфигурация считывается на уровне модуля

**Файл:** `app/modules/jina_reader.py`, строки 10–18

**Что плохо:**  
Все переменные конфигурации (`JINA_API_KEY`, `JINA_MAX_CHARS`, `JINA_TIMEOUT_SEC` и др.) читаются из `os.environ` один раз при импорте модуля. Это означает, что изменение переменных окружения во время работы процесса или в тестах (через `monkeypatch.setenv`) **не будет применено**, если модуль уже загружен. В частности, это объясняет, почему ключ Jina «остался» в окружении даже после его удаления из `.env`: модуль считал его при старте и держит значение в памяти.

**Почему это проблема:**  
Это ловушка для тестирования и для горячей перезагрузки конфигурации. Тест, который хочет проверить поведение без ключа Jina, не может просто убрать переменную — нужно перезагружать модуль целиком.

**Как исправить:**  
Перенести чтение конфигурации внутрь функций (`fetch_url` и `fetch_all`) через `os.environ.get(...)` в момент вызова. Альтернатива — использовать dependency injection через Pydantic Settings или параметры функций. Это стандартный паттерн для FastAPI-приложений.

---

### 2.2 `execution_engine.py`: Фильтрация доменов использует настройки Jina даже для Tavily

**Файл:** `app/modules/execution_engine.py`, строки 166–181

**Что плохо:**  
При дедупликации результатов в `execute_all` список исключаемых доменов берётся из переменной `JINA_SEARCH_EXCLUDE_DOMAINS`. Это семантически неверно: если используется Tavily как провайдер, фильтрация всё равно управляется параметрами с префиксом `JINA_*`. С ростом числа провайдеров это будет только запутывать конфигурацию.

**Как исправить:**  
Переименовать переменную в нейтральное `SEARCH_EXCLUDE_DOMAINS` как в `.env`, так и в коде. Это не ломает поведение, но делает конфигурацию провайдер-агностичной.

---

### 2.3 `redis_manager.py`: Гонка состояний в `_check_and_sync_fallback`

**Файл:** `app/modules/redis_manager.py`, строки 33–68

**Что плохо:**  
Функция проверяет `_redis_available` и затем пытается переключиться обратно на Redis. При этом `ping()` вызывается **вне** `_sync_lock`, а установка `_redis_available = True` — внутри. Однако между `ping()` и захватом лока несколько корутин могут одновременно пройти проверку. Двойная проверка `if _redis_available` внутри лока частично решает это, но `_check_and_sync_fallback` является глобальной переменной и меняется вне лока в `_handle_redis_error` без какой-либо защиты.

**Как исправить:**  
Использовать `asyncio.Lock` консистентно для всех операций изменения `_redis_available`: как при установке `False` (в `_handle_redis_error`), так и при установке `True` (в `_check_and_sync_fallback`). `ping()` тоже следует вынести внутрь лока или сделать отдельную «пробу» с собственным механизмом защиты.

---

### 2.4 `harvester.py`: Отключён, но код и зависимость от Redis остаются

**Файл:** `app/main.py`, строка 52; `app/modules/harvester.py`

**Что плохо:**  
Хаврестер отключён в `lifespan` комментарием «disabled as per task_011». Однако модуль `harvester.py` по-прежнему существует, Redis-клиент создаётся при старте (проблема 1.1), а `redis_manager` импортируется harvester'ом. На практике Redis не используется ни в одном активном пути выполнения кода, но его `REDIS_URL` обязательна для старта.

**Как исправить:**  
Если SearXNG и Redis планируются к использованию в будущем — добавить флаг `USE_REDIS=false` в `.env` и оборачивать инициализацию Redis этим флагом. Если функция временно не нужна — явно задокументировать это в коде (сейчас есть только комментарий без даты и причины).

## Раздел 3: Замечания среднего приоритета

---

### 3.1 `llm.py`: Singleton инициализируется при первом вызове любого модуля

**Файл:** `app/core/llm.py`, строки 138–148

**Что плохо:**  
`get_llm_provider()` использует паттерн `hasattr(_instance)`. При первом вызове создаётся `LLMProvider`, который сразу при инициализации обращается к API-ключу Gemini и устанавливает клиент. Если `GEMINI_API_KEY` не задан, конструктор упадёт не с понятным сообщением, а с исключением из глубины SDK. Кроме того, при создании используется `LLM_PROVIDER = os.environ.get(...)` на уровне модуля — снова та же проблема с «замороженной» конфигурацией (см. 2.1).

**Как исправить:**  
Добавить явную проверку наличия ключа перед созданием клиента с человекочитаемым `ValueError`. Вынести чтение `LLM_PROVIDER` внутрь `__init__` или `get_llm_provider()`.

---

### 3.2 `ai_judge.py`: Промпт загружается из файла при каждом вызове

**Файл:** `app/modules/ai_judge.py`, функция `build_system_prompt`, строки 50–68

**Что плохо:**  
При каждом вызове `judge()` функция открывает и читает `prompts.yaml` с диска (`with open(...)`). При нагрузке (несколько параллельных запросов или высокая частота) это тысячи лишних обращений к файловой системе. В отличие от `snippet_evaluator.py`, где файл читается один раз при загрузке модуля, здесь нет кеширования.

**Как исправить:**  
Прочитать `prompts.yaml` один раз при загрузке модуля (на уровне модуля, как сделано в `snippet_evaluator.py`), сохранить шаблон в переменную и применять форматирование строки только при сборке промпта.

---

### 3.3 `query_expansion.py`: Модуль не задействован в пайплайне

**Файл:** `app/modules/query_expansion.py`

**Что плохо:**  
Модуль реализован полностью, ссылка на него есть в `test_query_expansion.py`, но в `app/main.py` он не импортируется и не вызывается. На первой итерации используется точный запрос пользователя (это правильно), а на последующих — запросы генерирует `ai_judge`. `query_expansion.py` оказался между двух стульев.

**Почему это проблема:**  
Неясно, должен ли модуль использоваться или нет. Если нет — код захламляет проект. Если да — нужно явно решить, на каком этапе он применяется.

**Как исправить:**  
Либо удалить модуль и соответствующие тесты, либо явно интегрировать его в пайплайн с документированием роли (например, для предварительного обогащения первого итерационного запроса на сложных HARD-уровнях).

---

### 3.4 `execution_engine.py`: Создание `aiohttp.ClientSession` внутри каждого запроса

**Файл:** `app/modules/execution_engine.py`, строки 56 и 106

**Что плохо:**  
В `_search_jina` и `_search_tavily` создаётся новая `aiohttp.ClientSession()` на каждый поисковый вызов. При параллельном запуске 3–5 запросов одновременно (что является нормой при расширении запросов) создаётся 3–5 отдельных TCP-сессий. `ClientSession` рекомендуется создавать один раз и переиспользовать — это снижает накладные расходы на TCP handshake и повышает производительность. Та же проблема в `harvester.py`.

**Как исправить:**  
Передавать готовую `ClientSession` как параметр в функции поиска (аналогично тому, как это сделано в `jina_reader.py` через `fetch_all`). Сессию создавать один раз в `execute_all` и передавать вниз по цепочке.

---

### 3.5 `snippet_evaluator.py`: Промпт форматируется при импорте модуля

**Файл:** `app/modules/snippet_evaluator.py`, строки 23–29

**Что плохо:**  
`EVALUATOR_MAX_SELECTED` читается один раз при импорте и подставляется в промпт через `.format()`. Если переменная окружения изменится (например, в тесте через `monkeypatch`), промпт уже не обновится. Это та же проблема с «замороженной» конфигурацией.

**Как исправить:**  
Форматировать промпт непосредственно в функции `evaluate_snippets`, а не на уровне модуля.

---

### 3.6 `run_automated_tests.py`: Прямой импорт SDK провайдера в тест-раннере

**Файл:** `tests/run_automated_tests.py`, строки 46–47

**Что плохо:**  
Функция `synthesize_final_answer` напрямую импортирует `from google import genai` и создаёт клиент Gemini. Это нарушает принцип, заявленный в `GEMINI.md`: «Прямой импорт SDK провайдеров в бизнес-логике строго запрещён». Да, это тестовый код, а не бизнес-логика, но для синтеза ответов следовало бы переиспользовать уже готовый `app.core.llm`.

**Кроме того:** используется модель `gemini-3.1-flash-lite-preview`, которая хардкодом прописана в функции, — это нарушает правило «никаких хардкодных значений».

**Как исправить:**  
Заменить прямой вызов SDK на `await llm.generate_json(...)` из `app.core.llm`. Вынести имя модели синтезатора в отдельную переменную окружения, например `SYNTHESIZER_LLM_MODEL`.

## Раздел 4: Замечания низкого приоритета (технический долг)

---

### 4.1 `requirements.txt`: Нет закреплённых версий зависимостей

**Файл:** `requirements.txt`

**Что плохо:**  
Все зависимости указаны без версий (`fastapi`, `aiohttp` и т.д.). При установке на новом окружении или при обновлении pip может установить несовместимые версии библиотек, что сломает приложение без явных ошибок при установке.

**Как исправить:**  
Зафиксировать версии через `pip freeze > requirements.txt` или вручную указать минимальные версии в формате `fastapi>=0.111.0`. Для воспроизводимости окружения добавить `requirements-dev.txt` с зависимостями для разработки и тестирования (`pytest`, `pytest-asyncio`, `fakeredis` и др.), которые сейчас, судя по всему, доступны в среде, но не задокументированы.

---

### 4.2 `llm.py`: Двойной импорт `import time`

**Файл:** `app/core/llm.py`, строки 4 и 5

**Что плохо:**  
Модуль `time` импортируется дважды подряд (`import time` на строках 4 и 5). Это не вызывает ошибок (Python кеширует модули), но является явной опечаткой, сигнализирующей об отсутствии контроля качества.

**Как исправить:**  
Удалить дублирующую строку импорта.

---

### 4.3 `harvester.py`: Хардкод URL и задержки

**Файл:** `app/modules/harvester.py`, строки 12, 75

**Что плохо:**  
URL `https://searx.space/data/instances.json` жёстко прописан в коде, а не вынесен в переменную окружения. Задержка перед пингами (`await asyncio.sleep(0.5)`) также хардкодирована без возможности конфигурации извне.

**Как исправить:**  
Вынести URL каталога нод в `SEARXNG_INSTANCES_URL`, задержку пинга — в `HEALTH_CHECK_PING_DELAY_SEC`. Оба параметра добавить в `.env.example` с комментариями.

---

### 4.4 `search_router.py`: Jina Search не кодирует запрос в URL

**Файл:** `app/modules/search_router.py`, строка 21 (если модуль будет сохранён)

**Что плохо:**  
`url = f"https://s.jina.ai/{query}"` — запрос подставляется напрямую в URL без `urllib.parse.quote()`. Кириллица и спецсимволы приведут к некорректному URL и ошибке запроса. В `execution_engine.py` эта проблема уже исправлена (строка 24), но в старом модуле — нет.

**Как исправить:**  
После удаления `search_router.py` (см. п. 1.2) эта проблема исчезнет сама. Если модуль останется — добавить `urllib.parse.quote(query, safe="")`.
